# MiniNginx反向代理服务器实战：冰岩后端组25秋招实习

---

MiniNginx是一个基于 C++ 实现的轻量级 Web 服务器与反向代理项目，实现了HTTP 请求解析与响应构造、静态资源代理服务、反向代理转发、配置驱动路由、`epoll` 多路复用、配置热重载能力等核心功能。在这个README文档中，我将详细介绍各个阶段的核心任务实现，同时记录详细的工具使用和开发日志。




### 第0阶段：环境配置和工具链搭建（v0.1）

**目标**：完成开发工具和环境的搭建，打通开发链路与工程骨架，跑通第一个最小端到端服务。

**实现**：
- **搭建开发工具链（Linux/WSL + VSCode + Git + GitHub + CMake）**：
  - 在Windows上配置WSL2 Ubuntu环境，作为Linux开发环境。
  - 安装并配置VSCode及其Remote Development插件，实现远程开发的无缝体验。
  - 配置Git和GitHub，建立项目仓库并完成首次`push`。
  - 安装CMake，学习并编写了项目的初始`CMakeLists.txt`，定义项目名称、C++标准（C++11）和可执行文件目标。

- **初始化项目结构与 CMake 构建体系**：
  - 规划了项目目录：`src/`存放核心源代码，`config/`存放配置文件，`static/`存放静态资源，`build/`作为编译输出目录。
  - 在`src/`目录下创建了`main.cpp`、`server.cpp`、`http_parser.cpp`等核心文件的基础框架。
  - 通过`CMakeLists.txt`管理源文件依赖和链接库，确保构建过程清晰、可复现。

- **完成基本监听与连接建立流程**：
  - 在`server.cpp`中实现了一个简单的`TcpServer`类。
  - 使用`socket()`, `bind()`, `listen()`系统调用，成功创建了一个监听在`8080`端口的TCP socket。
  - 通过`accept()`接受客户端连接，并打印客户端的IP地址和端口，标志着第一个连接建立成功。

- **服务可响应基础 HTTP 请求**：
  - 在`accept()`后，通过`recv()`接收客户端发送的HTTP请求报文。
  - 手写了一个简易的响应构造函数，向客户端返回一个固定的“Hello, MiniNginx!”字符串。
  - 使用`curl http://localhost:8080`进行验证，成功看到了预期的响应，至此，一个最基础的HTTP服务跑通。

### 第1阶段：静态资源 + 反向代理（v1.0）

**目标**：实现最核心代理能力。

**实现**：
- **HTTP 报文解析/响应构造（手写）**：
  - 实现了`HttpRequest`类，通过解析请求行（方法、URI、版本）和请求头，提取出关键信息。
  - 实现了`HttpResponse`类，能根据业务逻辑（静态文件或代理）动态构造状态行、响应头和响应体。

- **静态资源服务能力**：
  - 实现了文件系统模块，根据请求的URI，在本地磁盘上查找对应的文件（如`static/index.html`）。
  - 读取文件内容，设置正确的`Content-Type`（通过文件扩展名映射，如`.html` -> `text/html`），并通过`HttpResponse`类返回给客户端。
  - 处理了文件不存在（404）的情况。

- **反向代理转发能力**：
  - 实现了`Upstream`模块，可以将特定请求（如以`/api/`开头）转发到配置好的上游服务器地址（如`127.0.0.1:9182`）。
  - 通过`socket`建立到上游服务器的连接，将原始请求转发过去，并接收上游的响应，最后将响应原样返回给客户端。

**关键修复与增强**：
- **上游代理地址调整**：开发调试时发现上游代理地址从历史默认值需要调整到实际运行地址：`127.0.0.1:9182`，修改后代理功能恢复正常。
- **静态路径映射问题**：为了兼容不同启动目录，将静态资源的根路径从绝对路径修改为相对于可执行文件路径的相对路径，通过获取`argv[0]`并计算目录，解决了路径查找失败的问题。
- **修复静态资源 404**：发现`/static/iamges`路径返回404，检查发现是`iamges`拼写错误，修正为`images`，并确保目录映射正确，解决了图片无法加载的问题。

### 第2阶段：配置驱动，实现路由匹配与请求转发（v2.0）

**目标**：从硬编码过渡到配置驱动。

**实现**：
- **自定义配置解析器（未依赖第三方库）**：
  - 设计了一套类似Nginx的简单配置语法，例如`server { listen 8080; location / { root /var/www; } }`。
  - 手写了一个配置解析器，通过逐行读取、词法分析和语法分析，将配置文件内容解析成一个树状结构的数据结构（如`Config`类包含多个`Server`对象，每个`Server`对象包含多个`Location`对象）。

- **支持解析与加载**：
  - 成功从配置文件中解析并加载了以下关键指令：
    - `worker_processes`：虽然尚未实现多进程，但解析了该值，为后续扩展做准备。
    - `http`、`server`、`location`：建立了配置的层级关系。
    - `listen`、`server_name`：用于决定哪个`server`块处理当前请求。
    - `proxy_pass`：用于反向代理转发。
    - `proxy_set_header Host`：支持在转发时修改请求头。
    - `root`、`index`：用于静态资源服务。

- **运行时按配置进行路由匹配与请求转发，而非硬编码路径**：
  - 服务器启动时加载配置文件，构建内部配置数据结构。
  - 当请求到达时，根据请求的`Host`头匹配`server`，再根据请求的URI匹配`location`。
  - 根据匹配到的`location`中的指令，决定是进行静态资源服务（`root`）还是反向代理转发（`proxy_pass`），彻底消除了v1.0版本中的硬编码逻辑。

### 第3阶段：整合运行 + 并发能力（v3.0）

**目标**：让阶段一与阶段二在高并发模型下协同工作。

**实现**：
- **服务端统一为 Linux 运行模式**：放弃了跨平台兼容的复杂性，聚焦于Linux平台，以便使用`epoll`等高级特性。

- **`socket + epoll` 多路复用事件循环**：
  - 将监听socket设置为非阻塞模式。
  - 创建一个`epoll`实例，并将监听socket注册到`epoll`上，监听`EPOLLIN`事件。
  - 在`while`主循环中，通过`epoll_wait()`获取就绪的事件列表。
  - 如果是监听socket有事件，则调用`accept()`接受新连接，并将新连接的socket也设置为非阻塞，注册到`epoll`上。
  - 如果是客户端socket有可读事件，则进行非阻塞的`recv()`，读取数据。

- **连接处理链路优化**：
  - **非阻塞+epoll驱动**：所有I/O操作都是非阻塞的，由`epoll`驱动，确保了单线程也能高效处理数千个并发连接。
  - **请求处理**：当读取到一个完整的HTTP请求后，立即调用第2阶段的路由匹配逻辑进行处理。
  - **响应发送**：处理完成后，将生成的响应通过`send()`返回给客户端。
  - **连接关闭**：响应发送后，关闭客户端连接，并从`epoll`中移除该socket。

- **配置热重载：基于文件修改时间检测变更，支持运行时重载路由配置**：
  - 实现了一个配置监控机制，定期检查配置文件的最后修改时间（`mtime`）。
  - 当检测到配置文件发生变化时，触发`reload_config()`函数。
  - `reload_config()`会重新加载并解析配置文件，生成新的配置数据结构。
  - 然后，原子地替换掉当前正在使用的配置数据结构指针，新到达的请求将使用新配置进行路由，实现了不停服的热重载。

**当前系统的功能边界**：
- `master-worker` 的完整生产级进程模型（`fork + worker 管理 + 平滑重启`）尚未完全落地。目前是单进程架构，但已通过`epoll`实现了高并发，热重载能力也已就绪。完整的进程管理模型作为未来的优化方向。

### 第4阶段：前端展示与交互体验（v3.1）

**目标**：提高演示效果与产品化观感。

**本轮完成**：
- **根路径首页 `http://localhost:8080/`**：
  - **UI微调**：修复了“查看完整欢迎页”按钮的视觉问题。将默认背景色调整为更浅的灰色，仅在鼠标悬停时变为深色，提升了视觉友好度，避免了默认态颜色过深带来的突兀感。
  - **全面视觉重构**：对`static/index.html`进行了彻底的重写，引入了现代Web前端设计风格。页面主体为一个悬浮的3D卡片，增加了视觉层次感；同时引入了背景渐变流动动画，营造科技感和动感。
  - **交互与动画**：
    - 为3D卡片实现了`tilt`倾斜交互效果，鼠标移动时卡片会跟随倾斜，增强了趣味性和互动性。
    - 实现了页面分段入场动画，文字、按钮、卡片等元素会依次以淡入和上浮的方式出现，让页面加载过程不再生硬。
    - 加入了暗/亮主题切换功能，并利用`localStorage`进行持久化，用户下次访问时会自动记住其偏好。
  - **细节优化**：根据最终要求移除了鼠标跟随光晕效果，使页面看起来更简洁。同时轻微加快了背景渐变流动的节奏，使观感更灵动但又不会过快分散用户注意力。

- **仓库工程化补充**：
  - 新增并完善了`.gitignore`文件，现在会忽略`build/`目录下的所有构建产物（如`*.o`, `MiniNginx`可执行文件）、VSCode的本地配置（`.vscode/`）、以及临时交换文件（`*.swp`），确保了仓库的干净整洁。

## 4. 常用工具的命令行

### 4.1 本地开发与运行（CMake）

| 场景 | 命令顺序 | 说明 |
|---|---|---|
| 首次构建 | `mkdir -p build && cd build` → `cmake ..` → `make` → `./MiniNginx` | 初始化构建并启动服务 |
| 日常增量编译 | `cd build` → `make` → `./MiniNginx` | 修改代码后的快速验证 |
| 全量重构建 | `cd build` → `rm -rf *` → `cmake ..` → `make` → `./MiniNginx` | CMake 配置变更或构建异常时使用 |
| 接口验证 | `curl -i http://localhost:8080` | 验证首页 |
| 静态资源验证 | `curl -i http://localhost:8080/static/index.html` | 验证静态映射 |
| 代理验证 | `curl -i http://localhost:8080/api/v1/healthy` | 验证反向代理 |
| 结束进程 | `pkill -f MiniNginx` | 停止当前服务 |

### 4.2 Git 提交流程（推荐顺序）

| 步骤 | 命令 | 目的 |

### 4.3 日常开发流程

当时的日常开发流程：

1. 更改源代码
2. 进行编译运行

```bash
cd build
make
./MiniNginx
# 新开终端测试
curl http://localhost:8080
cd ..
git add .
git commit -m "feat: ..."
git pull origin main   # 推送前再次同步，确保无冲突
git push origin main
pkill -f MiniNginx
```

如需重新 CMake 全量构建：

```bash
cd build
rm -rf *
cmake ..
make
./MiniNginx
```
